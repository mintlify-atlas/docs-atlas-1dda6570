Contributions are welcome and will be fully credited. Please read and understand this guide before creating an issue or pull request.

## Etiquette

This project is open source, and as such, the maintainers give their free time to build and maintain the source code. They make the code freely available in the hope that it will be of use to other developers.

Please be considerate towards maintainers when raising issues or presenting pull requests. Let's show the world that developers are civilized and selfless people.

<Note>
It's the duty of the maintainer to ensure that all submissions to the project are of sufficient quality to benefit the project. Many developers have different skillsets, strengths, and weaknesses. Respect the maintainer's decision, and do not be upset or abusive if your submission is not used.
</Note>

## Viability

When requesting or submitting new features, first consider whether it might be useful to others. Open source projects are used by many developers, who may have entirely different needs to your own. Think about whether or not your feature is likely to be used by other users of the project.

## Procedure

### Before filing an issue

<Steps>
  <Step title="Replicate the problem">
    Attempt to replicate the problem to ensure that it wasn't a coincidental incident.
  </Step>
  <Step title="Check existing features">
    Check to make sure your feature suggestion isn't already present within the project.
  </Step>
  <Step title="Review pull requests">
    Check the pull requests tab to ensure that the bug doesn't have a fix in progress, or that the feature isn't already being developed.
  </Step>
</Steps>

### Before submitting a pull request

<Steps>
  <Step title="Check the codebase">
    Ensure that your feature doesn't already exist in the codebase.
  </Step>
  <Step title="Review existing PRs">
    Check the pull requests to ensure that another person hasn't already submitted the feature or fix.
  </Step>
</Steps>

## Requirements

If the project maintainer has any additional requirements, you will find them listed here.

<AccordionGroup>
  <Accordion title="Coding standards">
    Follow the [PSR-2 Coding Standard](https://github.com/php-fig/fig-standards/blob/master/accepted/PSR-2-coding-style-guide.md). The easiest way to apply the conventions is to install [PHP Code Sniffer](http://pear.php.net/package/PHP_CodeSniffer).
  </Accordion>

  <Accordion title="Testing">
    Add tests! Your patch won't be accepted if it doesn't have tests.
  </Accordion>

  <Accordion title="Documentation">
    Document any change in behavior. Make sure the `README.md` and any other relevant documentation are kept up-to-date.
  </Accordion>

  <Accordion title="Release cycle">
    Consider our release cycle. We try to follow [SemVer v2.0.0](http://semver.org/). Randomly breaking public APIs is not an option.
  </Accordion>

  <Accordion title="One PR per feature">
    If you want to do more than one thing, send multiple pull requests.
  </Accordion>

  <Accordion title="Coherent history">
    Send coherent history. Make sure each individual commit in your pull request is meaningful. If you had to make multiple intermediate commits while developing, please [squash them](http://www.git-scm.com/book/en/v2/Git-Tools-Rewriting-History#Changing-Multiple-Commit-Messages) before submitting.
  </Accordion>
</AccordionGroup>
